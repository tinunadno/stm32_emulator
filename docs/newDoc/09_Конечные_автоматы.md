# Описание системы и модулей как конечных автоматов

## 1. Система в целом — Simulator

Simulator является главным конечным автоматом, который оркестрирует все остальные.

**Состояния:**
- `UNINITIALIZED` — начальное, до вызова `simulator_init()`
- `INITIALIZED` — все подсистемы созданы, бинарник не загружен
- `READY` — бинарник загружен во Flash, PC установлен на вектор Reset
- `RUNNING` — идёт пошаговое выполнение (`simulator_step()`)
- `HALTED` — выполнение остановлено (брейкпоинт, GDB-команда, ошибка)

**Переходы:**
```
UNINITIALIZED --[simulator_init()]--> INITIALIZED
INITIALIZED   --[simulator_load()]--> READY
READY         --[старт/continue]  --> RUNNING
RUNNING       --[breakpoint hit]  --> HALTED
RUNNING       --[invalid instr.]  --> HALTED
RUNNING       --[GDB halt (^C)]   --> HALTED
HALTED        --[GDB continue]    --> RUNNING
```

**Один шаг RUNNING** (`simulator_step()`) — это микроцикл:
1. Инкремент счётчика цикла
2. Tick всех периферийных устройств
3. Диспатч событий из EventQueue (если `cycle >= event.when`)
4. Выполнение одной инструкции в Core
5. Проверка брейкпоинтов
6. Проверка NVIC — если есть ожидающее прерывание с достаточным приоритетом, вход в ISR

## 2. Core (CPU — Thumb-2 интерпретатор)

**Состояния:**
- `FETCH` — читается 16-битный полуслов с адреса PC
- `FETCH_WIDE` — после обнаружения 32-битной инструкции (старший полубайт первого полуслова = 0b111xx) читается второй полуслов
- `DECODE` — маскирование и классификация опкода
- `EXECUTE` — выполнение операции, обновление флагов APSR
- `EXCEPTION_ENTRY` — при наступлении прерывания: сохранение контекста (r0–r3, r12, LR, PC, xPSR) в стек, переход к вектору ISR
- `EXCEPTION_EXIT` — при `BX LR` с EXC_RETURN: восстановление контекста из стека, возврат к прерванному коду

**Переходы:**
```
FETCH       --[обычная инструкция (16 бит)]       --> DECODE
FETCH       --[32-битный префикс]                  --> FETCH_WIDE
FETCH_WIDE  --[второй полуслов получен]             --> DECODE
DECODE      --[опкод распознан]                    --> EXECUTE
DECODE      --[неизвестный опкод]                  --> (STATUS_INVALID_INSTRUCTION)
EXECUTE     --[нормально]                          --> FETCH (следующий цикл)
EXECUTE     --[SVC / HardFault / внешний IRQ]      --> EXCEPTION_ENTRY
EXCEPTION_ENTRY --[вектор получен, стек сохранён]  --> FETCH (PC = вектор ISR)
EXCEPTION_EXIT  --[EXC_RETURN, стек восстановлен]  --> FETCH (PC = адрес возврата)
```

**Внутренний регистровый файл:** r0–r15, xPSR (N/Z/C/V, IPSR), `current_irq`, `interruptible`.

## 3. Bus (маршрутизатор адресного пространства)

Bus — это не классический автомат с состояниями, но его поведение можно описать как автомат запрос/ответ.

**Состояния:**
- `IDLE` — ожидание обращения
- `ROUTING` — линейный поиск подходящего `BusRegion` по адресу
- `DISPATCHING` — вызов vtable (`read` или `write`) найденного региона
- `FAULT` — адрес не найден ни в одном регионе → `STATUS_INVALID_ADDRESS`

**Переходы:**
```
IDLE        --[bus_read(addr) / bus_write(addr)]  --> ROUTING
ROUTING     --[region найден]                     --> DISPATCHING
ROUTING     --[region не найден]                  --> FAULT
DISPATCHING --[vtable вернул результат]            --> IDLE
FAULT       --                                    --> IDLE (с кодом ошибки)
```

## 4. EventQueue (планировщик событий)

**Состояния:**
- `EMPTY` — очередь пуста (нет запланированных событий)
- `WAITING` — в очереди есть события, но `now < event[0].when`
- `FIRING` — `now >= event[0].when`: callback вызывается, событие извлекается из кучи

**Переходы:**
```
EMPTY   --[event_queue_schedule()]  --> WAITING
WAITING --[schedule новое]          --> WAITING (перестройка кучи)
WAITING --[dispatch, now < when]    --> WAITING
WAITING --[dispatch, now >= when]   --> FIRING
FIRING  --[callback выполнен]       --> EMPTY или WAITING (если осталось)
```

**Механизм Generation ID:** каждое событие имеет `id`. Если периферия перепланирует таймер (например, ARR изменился), старый callback обнаруживает несовпадение ID и тихо игнорирует вызов — стейл-событие не приводит к неправильному срабатыванию.

## 5. TIM2 (General Purpose Timer)

**Состояния:**
- `DISABLED` — CEN = 0, счёт не идёт
- `RUNNING` — CEN = 1, счёт идёт (ленивый: CNT вычисляется на чтение)
- `OVERFLOW_PENDING` — событие overflow сработало, UIF = 1
- `IRQ_PENDING` — UIE = 1 → прерывание IRQ28 выставлено в NVIC

**Переходы:**
```
DISABLED       --[запись CEN=1]           --> RUNNING (планируется overflow event)
RUNNING        --[запись CEN=0]           --> DISABLED (event становится stale по gen ID)
RUNNING        --[overflow event fired]   --> OVERFLOW_PENDING
OVERFLOW_PENDING --[UIE=1]               --> IRQ_PENDING (nvic_set_pending(28))
OVERFLOW_PENDING --[UIE=0]               --> RUNNING (без прерывания)
IRQ_PENDING    --[firmware читает SR]     --> RUNNING (firmware сбрасывает UIF)
```

**Ленивый CNT:** `CNT = (elapsed_cycles / (PSC+1)) % ARR`. Физически не инкрементируется каждый цикл — вычисляется в момент чтения регистра.

## 6. SysTick

Аналогичен TIM2, но используется как системный таймер Cortex-M3.

**Состояния:**
- `DISABLED` — ENABLE = 0
- `COUNTING` — ENABLE = 1, 24-битный обратный счётчик (ленивый CVR)
- `WRAP_PENDING` — CVR достиг 0, COUNTFLAG = 1
- `EXCEPTION_PENDING` — TICKINT = 1 → выставляется исключение #15

**Ленивый CVR:** `CVR = RVR - (elapsed_cycles % (RVR+1))`.

**Переходы:**
```
DISABLED      --[ENABLE=1]            --> COUNTING (планируется wrap event)
COUNTING      --[ENABLE=0]            --> DISABLED
COUNTING      --[wrap event fired]    --> WRAP_PENDING
WRAP_PENDING  --[TICKINT=1]           --> EXCEPTION_PENDING (exception #15)
EXCEPTION_PENDING --[ISR завершён]    --> COUNTING (перепланируется wrap event)
```

## 7. UART (USART1)

UART имеет два независимых автомата: TX и RX.

### TX-автомат:
- `TX_IDLE` — DR не записан
- `TX_PENDING` — firmware записал байт в DR, `tx_pending = 1`
- `TX_TRANSMITTING` — `uart_tick()` обнаружил `tx_pending`: вызов `output_fn`, сброс флага, TXE = 1
- `TX_IRQ` — если TXEIE = 1 → IRQ37 выставляется в NVIC

```
TX_IDLE       --[запись DR]          --> TX_PENDING
TX_PENDING    --[uart_tick()]        --> TX_TRANSMITTING
TX_TRANSMITTING --[output_fn()]      --> TX_IDLE (TXE=1)
TX_IDLE       --[TXEIE=1]            --> TX_IRQ (если TXE=1)
```

### RX-автомат:
- `RX_IDLE` — нет входящих данных
- `RX_BUFFERED` — `uart_incoming_char()` положил байт в кольцевой буфер, RXNE = 1
- `RX_IRQ` — если RXNEIE = 1 → IRQ37

```
RX_IDLE      --[uart_incoming_char()] --> RX_BUFFERED (RXNE=1)
RX_BUFFERED  --[firmware читает DR]   --> RX_IDLE (RXNE=0)
RX_BUFFERED  --[RXNEIE=1]             --> RX_IRQ
```

## 8. NVIC (контроллер прерываний)

**Состояния на уровне каждого IRQ (43 линии):**
- `INACTIVE` — pending=0, active=0
- `PENDING` — pending=1, active=0 (прерывание запрошено, но ещё не обслуживается)
- `ACTIVE` — pending=0, active=1 (ISR выполняется)
- `ACTIVE_PENDING` — pending=1, active=1 (ISR выполняется, новый запрос пришёл)

**Переходы:**
```
INACTIVE     --[nvic_set_pending(n)]         --> PENDING
PENDING      --[Core входит в ISR]           --> ACTIVE
ACTIVE       --[nvic_set_pending(n)]         --> ACTIVE_PENDING
ACTIVE       --[ISR завершён (EXC_RETURN)]   --> INACTIVE
ACTIVE_PENDING --[ISR завершён]              --> PENDING (будет вызван снова)
PENDING      --[nvic_clear_pending(n)]       --> INACTIVE
```

**Приоритет:** `nvic_get_pending_irq()` возвращает IRQ с наименьшим числовым значением приоритета среди enabled+pending, только если его приоритет выше текущего активного прерывания. Это обеспечивает вытеснение (preemption).

## 9. GDB Stub

**Состояния:**
- `LISTENING` — TCP сервер ждёт подключения
- `CONNECTED` — клиент (GDB) подключился
- `RECEIVING` — читаем пакет `$data#checksum`
- `PROCESSING` — разбираем команду (`g`, `G`, `m`, `M`, `c`, `s`, `Z`, `z`, `?`, `k`)
- `SENDING` — отправляем ответ `+$reply#checksum`
- `DISCONNECTED` — клиент отключился или `k` команда

```
LISTENING    --[accept()]                      --> CONNECTED
CONNECTED    --[recv 0x03 (Ctrl+C)]            --> PROCESSING (halt)
CONNECTED    --[recv '$']                      --> RECEIVING
RECEIVING    --[получен '#'+checksum]          --> PROCESSING
PROCESSING   --[команда 'c']                   --> SENDING (resume sim)
PROCESSING   --[команда 's']                   --> SENDING (step sim, then halt)
PROCESSING   --[другие команды]                --> SENDING
SENDING      --[ответ отправлен]               --> CONNECTED
CONNECTED    --[закрытие соединения]           --> LISTENING
```

Simulator в режиме GDB работает иначе: `gdb_stub_run()` сам управляет циклом выполнения (вызывает `simulator_step()` в цикле при `continue`), а не наоборот.

---

## 10. Debugger (брейкпоинты)

Простой автомат поиска:

**Состояния:**
- `IDLE` — список брейкпоинтов не содержит текущего PC
- `HIT` — текущий PC совпал с адресом в списке

```
IDLE  --[debugger_check(PC) == bp_addr]  --> HIT
HIT   --[sim->halted = 1]                --> IDLE (ждём следующего шага)
```

Брейкпоинты добавляются/удаляются через GDB-команды `Z0` / `z0`.

---

## 11. GPIO

Каждый пин — это независимый двух-состоянийный автомат конфигурации.

**Состояния пина:**
- `INPUT` — режим входа (IDR читает `ext_input`)
- `OUTPUT` — режим выхода (IDR отражает ODR)
- `ALTERNATE` — управление периферией (UART, TIM)
- `ANALOG` — аналоговый режим (не используется в эмуляторе)

**Настройка:** через регистры CRL (пины 0–7) и CRH (пины 8–15) — 4 бита на пин (MODE + CNF).

**Запись:** BSRR/BRR обеспечивают атомарный set/reset бит ODR без операции read-modify-write.

## Взаимодействие автоматов (сводная схема)

```
firmware (машинный код)
        |
        v
   [Core: FETCH→DECODE→EXECUTE]
        |                   |
        | bus_read/write     | exception entry/exit
        v                   v
   [Bus: ROUTING]        [NVIC: PENDING→ACTIVE]
        |                   ^
   ┌────┼────────────────────┼──────────────┐
   |    |                   |              |
   v    v                   |              v
[Flash] [SRAM]         [NVIC regs]    [TIM2 / SysTick]
                                           |
                       [EventQueue] <------+
                            |
                       overflow event fired
                            |
                            v
                       nvic_set_pending()
                            |
                            v
                    [NVIC: PENDING state]
                            |
              (next simulator_step check)
                            |
                            v
                   [Core: EXCEPTION_ENTRY]
```

Ключевое: **EventQueue** — единственный путь, по которому TIM2 и SysTick порождают прерывания. Периферия не проверяется каждый цикл; вместо этого она планирует событие на будущий цикл, что делает систему эффективной даже при больших значениях PSC/ARR.
