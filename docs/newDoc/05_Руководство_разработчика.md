# Руководство разработчика

## 1. Назначение документа

Документ фиксирует правила развития кодовой базы симулятора, описание точек расширения и требования к качеству изменений.

## 2. Структура проекта

- `src/main.c` - точка входа и разбор параметров запуска.
- `src/simulator` - orchestration layer.
- `src/core` - CPU interpreter и exceptions.
- `src/memory`, `src/bus`, `src/nvic` - базовые платформенные подсистемы.
- `src/peripherals` - периферийные устройства.
- `src/gdb_stub` - RSP сервер.
- `src/ui` - интерактивный интерфейс.
- `src/tests` - unit/integration тесты.

## 3. Сборка и тестирование

Рабочие команды (из `src`):

```bash
make
make test
make clean
```

## 4. Архитектурные контракты

## 4.1 Контракты модуля Core

- `core_step` выполняет ровно одну инструкцию (16-бит или 32-бит).
- Core не обращается к памяти напрямую, только через `bus_read/bus_write`.
- Логика IRQ опирается на API `nvic_get_pending_irq`, `acknowledge`, `complete`.

## 4.2 Контракты Bus

- каждый регион обязан иметь корректные `read/write` callback;
- допускается регистрация до `BUS_MAX_REGIONS`;
- обратная совместимость по формату `BusRegion` обязательна для всех периферий.

## 4.3 Контракты Simulator

- последовательность `tick -> core_step -> debugger_check` не должна нарушаться;
- reset обязан приводить модули к согласованному стартовому состоянию;
- новый peripheral регистрируется через `simulator_add_peripheral`.

## 5. Расширение: добавление новой периферии

1. Создать структуру состояния периферии.
2. Реализовать callbacks:
   - `read`
   - `write`
   - `tick` (если требуется временная динамика)
   - `reset`
3. Сформировать `Peripheral` через helper `*_as_peripheral`.
4. В `simulator_init`:
   - инициализировать модуль;
   - зарегистрировать регион в Bus;
   - добавить в список tickable устройств.
5. Создать unit tests для read/write/tick/reset.
6. Добавить интеграционный тест с участием CPU/NVIC (если используется IRQ).
7. Обновить документацию:
   - [[02_Архитектура_системы|Архитектура системы]];
   - [[03_Описание_модулей|Описание модулей]];
   - [[06_Программа_и_методика_испытаний|Программа и методика испытаний]].

## 6. Требования к изменениям в Core

- любые изменения декодирования должны сопровождаться тест-кейсами;
- при добавлении инструкций обновлять таблицу поддерживаемых классов в документации;
- для новых 32-битных инструкций контролировать корректность инкремента `PC`;
- нарушение поведения `xPSR` флагов считается критическим дефектом.

## 7. Работа с GDB Stub

При изменениях в RSP:

- сохранять обратную совместимость с базовыми командами (`g/G`, `m/M`, `c/s`, `Z/z`);
- валидировать checksums и ACK/NACK-логику;
- держать корректным target description XML для Cortex-M профиля;
- тестировать подключение в `arm-none-eabi-gdb` и через расширения IDE.

## 8. Код-стайл и инженерные практики

- язык: C11;
- обязательные флаги компиляции предупреждений;
- минимизация глобальных переменных;
- комментарии только там, где логика действительно нетривиальна;
- избегать неявных контрактов между модулями.

## 9. Тестовая стратегия

- unit tests на каждый отдельный модуль;
- integration tests на сквозные сценарии:
  - прерывание по таймеру;
  - остановка на breakpoint;
  - UART output через callback;
- при багфиксе сначала добавлять/обновлять тест, затем править реализацию.

## 10. Рекомендованный шаблон PR/изменения

1. Проблема или цель.
2. Что изменено (по модулям).
3. Риски регрессии.
4. Как проверено (`make`, `make test`, ручные сценарии).
5. Обновленные документы.

## 11. Известные технические ограничения

- ISA покрыт частично;
- отсутствует ELF loader;
- отсутствует точное аппаратное моделирование временных характеристик;
- модель периферии минимальная.
