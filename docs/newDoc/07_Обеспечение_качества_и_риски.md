# Обеспечение качества и риски

## 1. Цель документа

Определить подход к качеству программного продукта, процессам верификации и управлению рисками разработки симулятора STM32.

## 2. Политика качества проекта

Качество обеспечивается совокупностью мер:

- проектирование модульной архитектуры с явными интерфейсами;
- обязательное автоматическое тестирование;
- код-ревью изменений;
- поддержание актуальной технической документации;
- проверка пользовательских сценариев перед демонстрацией.

## 3. Метрики качества

### 3.1 Функциональная полнота

- доля требований из [[01_Техническое_задание|ТЗ]], покрытых реализацией;
- доля требований, подтвержденных тестами.

### 3.2 Тестовая зрелость

- количество проходящих unit/integration тестов;
- отсутствие нестабильных (flaky) тестов;
- наличие теста на каждый критичный багфикc.

### 3.3 Эксплуатационная устойчивость

- отсутствие аварий в стандартных CLI-сценариях;
- корректная обработка ошибок ввода и адресных нарушений;
- стабильность GDB-сессии при базовых командах.

## 4. Процесс контроля качества

1. **Разработка:** изменение кода в отдельной ветке/коммите.
2. **Самопроверка:** локальная сборка и `make test`.
3. **Документация:** актуализация документов при изменении поведения.
4. **Ревью:** проверка архитектурных и тестовых последствий.
5. **Интеграция:** объединение только после подтверждения критериев.

## 5. Реестр рисков

| ID | Риск | Вероятность | Влияние | Мероприятия |
|---|---|---|---|---|
| R1 | Некорректная эмуляция инструкций | Средняя | Высокое | расширять тесты `core`, вводить regression tests |
| R2 | Ошибки обработки IRQ | Средняя | Высокое | интеграционные тесты на exception entry/exit |
| R3 | Регрессии при добавлении периферии | Средняя | Среднее | контракт `Peripheral`, изоляция модулей |
| R4 | Несовместимость с GDB-клиентами | Низкая | Высокое | проверка ключевых RSP-команд и target XML |
| R5 | Рассинхронизация кода и документации | Средняя | Среднее | обязательный doc-checklist в ревью |
| R6 | Ограниченность ISA | Высокая | Среднее | фиксировать покрытие и границы применимости |

## 6. План реагирования на риски

- для высоких рисков (R1, R2, R4) - приоритетное тестирование на каждом релиз-кандидате;
- для средних рисков - контроль через ревью и регулярные smoke-тесты;
- для документарных рисков - назначение ответственного за комплект документов.

## 7. Управление дефектами

Классы дефектов:

- **Критический:** падение программы, потеря управляемости, невозможность запуска;
- **Высокий:** неверная логика исполнения/IRQ/GDB, влияющая на результат моделирования;
- **Средний:** частичная недоступность функции при наличии обхода;
- **Низкий:** косметические несоответствия UI/логов/документации.

Правила:

- критические и высокие дефекты блокируют демонстрационную поставку;
- после исправления обязателен тест, воспроизводящий проблему;
- закрытие дефекта без проверки тестом не допускается.

## 8. Релизная готовность

Минимальные критерии:

- успешная сборка;
- прохождение автоматических тестов;
- прохождение ручных сценариев CLI/GDB;
- актуальная документация из `docs/newDoc`;
- отсутствие открытых дефектов уровня критический/высокий.

## 9. Улучшения качества (дорожная карта)

- расширить покрытие ISA и таблицу поддерживаемых инструкций;
- добавить стенд для сравнения поведения с реальным железом в эталонных сценариях;
- внедрить статический анализ и форматирование в CI;
- добавить автоматический отчет трассируемости "требование -> тест".
