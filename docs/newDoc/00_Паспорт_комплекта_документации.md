# Паспорт комплекта документации

## 1. Сведения о проекте

- **Наименование:** программный симулятор микроконтроллера STM32F103C8T6.
- **Тип изделия:** учебно-исследовательское программное средство моделирования.
- **Назначение:** выполнение бинарных прошивок и отладка логики работы ядра Cortex-M3 и базовой периферии без физической платы.
- **Среда реализации:** язык C, модульная архитектура, консольный интерфейс, GDB RSP.

## 2. Основание для разработки документации

Документация подготовлена для учебного проекта команды разработчиков как комплект, приближенный к структуре документов по ГОСТ 19/34:

- техническое задание;
- описание программы и архитектуры;
- руководство пользователя;
- руководство программиста;
- программа и методика испытаний;
- пояснительные и организационные материалы.

## 4. Правила ведения документации в Obsidian

- Каждый файл является самостоятельным документом.
- Междокументные связи оформляются Obsidian-ссылками `[[...]]`.
- Изменения в программном коде сопровождаются актуализацией минимум трех документов:
  - `02_Архитектура_системы`;
  - `03_Описание_модулей`;
  - `06_Программа_и_методика_испытаний`.
- При изменении внешних интерфейсов обязательно обновляются:
  - пользовательские сценарии в `04_Руководство_пользователя`;
  - API и расширяемость в `05_Руководство_разработчика`.

## 5. Принятые соглашения

- **Термин "система"** обозначает эмулятор в целом.
- **Термин "модуль"** обозначает отдельный компонент из каталога `src`.
- **Статус-коды** соответствуют перечислению `Status` (`STATUS_OK`, `STATUS_ERROR` и др.).
- Адреса, смещения и регистры приводятся в шестнадцатеричном формате.
- Версия документации должна быть синхронизирована с актуальным состоянием ветки репозитория.

## 6. Матрица трассируемости (верхний уровень)

| Требование | Реализация | Подтверждение |
|---|---|---|
| Исполнение инструкций ядра | `src/core` | модульные тесты `test_core` |
| Память Flash/SRAM | `src/memory` | `test_memory`, `test_bus` |
| Маршрутизация MMIO | `src/bus` | `test_bus`, интеграционные тесты |
| Прерывания NVIC | `src/nvic` + `core` | `test_nvic`, `test_integration` |
| Таймер TIM2 | `src/peripherals/timer` | `test_timer`, `test_integration` |
| SysTick | `src/peripherals/systick` | `test_systick` |
| RCC | `src/peripherals/rcc` | `test_rcc` |
| GPIO (A/B/C) | `src/peripherals/gpio` | `test_gpio` |
| UART USART1 + логгер | `src/peripherals/uart` | `test_uart`, `test_integration` |
| Очередь событий | `src/events/event_queue` | косвенно `test_timer`, `test_systick` |
| NVIC bus-периферия | `src/nvic/nvic_bus` | `test_nvic`, интеграционные проверки |
| Отладка breakpoints | `src/debugger` | `test_debugger` |
| Внешняя отладка GDB | `src/gdb_stub` | ручная проверка GDB RSP |

## 7. Порядок согласования

1. Разработчик обновляет код и документы в одной ветке.
2. Ответственный за архитектуру проверяет:
   - непротиворечивость требований и реализации;
   - актуальность таблиц адресов и интерфейсов.
3. Ответственный за тестирование подтверждает покрытие измененных требований.
4. После командной проверки комплект считается готовым к защите проекта.
